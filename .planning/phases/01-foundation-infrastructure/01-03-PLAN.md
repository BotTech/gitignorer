<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User/Agent → Commander.js | Command-line arguments and flags (untrusted) |
| Commander.js → Command handlers | Parsed and validated options (trusted after validation) |
| Command handlers → Formatters | Validated result objects (trusted) |
| Formatters → stdout/stderr | Formatted output (data to stdout, messages to stderr) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Command handler validation | mitigate | Each command validates input via Zod schema before any processing (CONTEXT.md D-12). Invalid input throws ValidationError before business logic. |
| T-03-02 | Information Disclosure | configureOutput separation | mitigate | program.configureOutput() enforces stdout/stderr separation. Data never written to stderr, messages never written to stdout. |
| T-03-03 | Spoofing | Command registration | accept | Commands are registered at startup, not modified at runtime. No authentication needed (local CLI). |
| T-03-04 | Denial of Service | Blocking async handlers | mitigate | All command handlers are async (use async keyword). parseAsync() ensures non-blocking execution. No synchronous file operations in command flow. |
| T-03-05 | Tampering | Flag injection via command options | mitigate | applyCommonFlags() only accepts predefined flags. Commander.js validates flag names. User can't inject arbitrary flags. |
| T-03-06 | Information Disclosure | Error output in handlers | mitigate | Commands throw typed errors (ValidationError, etc.). Error handler (Plan 01) formats as JSON without leaking sensitive data. |
| T-03-07 | Elevation of Privilege | N/A | accept | No privilege escalation in command framework. Commands run with user's existing permissions. |
</threat_model>
