<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → Zod schemas | Command-line flags, JSON payloads, template names (untrusted) |
| Stdin → parseStdinJSON | Piped JSON data (untrusted) |
| Zod refine() → ValidationError | Custom validation logic (trusted) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | dangerousPatternSchema | mitigate | Zod refine() rejects .. (path traversal), control chars (< 0x20), and query injection chars (? # %). Validation happens BEFORE any file system operations. |
| T-02-02 | Tampering | parseStdinJSON | mitigate | Non-blocking async parser prevents DoS via pipe buffer saturation. JSON.parse() validates structure before use. Malformed JSON throws ValidationError, not raw error. |
| T-02-03 | Information Disclosure | error.toJSON() | mitigate | Error classes only expose safe properties (message, code, validated context). No raw user input, no stack traces, no file system paths in ValidationError. |
| T-02-04 | Spoofing | parseOutputFlag/parseInputFlag | accept | Flag parsing validates enum values but doesn't handle authentication/authorization (not applicable for Phase 1). |
| T-02-05 | Tampering | jsonInputSchema | mitigate | Schema enforces enum for 'command' field, preventing command injection. Only valid command names accepted. |
| T-02-06 | Denial of Service | parseStdinJSON async loop | mitigate | Using for await (async iterator) instead of synchronous reads prevents event loop blocking. Large inputs handled as stream, not loaded all at once. |
</threat_model>
